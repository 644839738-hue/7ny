# Product Requirements Document — SpriteForge AI

## 项目定位

SpriteForge AI **不是普通的文生图工具**。市面上的 AI 图像生成器（DALL·E、Midjourney、Stable Diffusion）产出的图像无法直接用于游戏引擎——它们缺少透明背景、尺寸不标准、无 Sprite Sheet 元数据、Tile 边缘可能不连续。SpriteForge AI 填补了「AI 生成图像」到「游戏引擎可直接导入的资产」之间的工程化空白。

**一句话定位**：面向 2D 游戏开发工作流的 AI 素材生成与资产化管线。

## 项目背景与痛点

独立游戏开发者和小型 2D 游戏团队面临以下困境：

| 痛点 | 现状 | SpriteForge AI 方案 |
|------|------|---------------------|
| 美术资源匮乏 | 买素材包（风格不统一）或外包（贵、慢） | AI 生成 + 参数约束，风格可控 |
| 资产处理繁琐 | 手动去背景、调尺寸、对齐、写元数据 | 一键自动资产化管线 |
| Sprite Sheet 制作 | 手动拼接 + 计算帧坐标 | 自动拼接 + JSON 元数据输出 |
| Tile 无缝检查 | 肉眼检查，易遗漏 | 算法自动检测边缘一致性 |
| 引擎适配 | 不同引擎对素材格式要求不同 | 支持 Unity / Godot 格式导出 |

## 目标用户

- **独立游戏开发者**：一人兼顾程序+美术，需要快速产出可用素材
- **小型 2D 游戏团队（2-5 人）**：有程序但缺美术，或美术产能不足
- **Game Jam 参赛者**：48 小时内需要大量风格统一的 2D 素材

## 核心需求（MoSCoW 分级）

### Must Have（P0 — MVP 必须实现）

| ID | 需求 | 说明 |
|----|------|------|
| R1 | 文本/参数生成素材 | 用户输入 prompt + 选择类型/尺寸/风格，生成对应素材 |
| R2 | 四类素材支持 | 角色（Character）、道具（Prop）、Tile、UI |
| R3 | 像素风双尺寸 | 32×32 和 64×64 两档 |
| R4 | DEMO 模式 | 无外部 AI API 时，用内置样例素材模拟完整生成流程 |
| R5 | 透明背景处理 | 自动移除生成图像的背景，输出 RGBA PNG |
| R6 | 尺寸标准化 | 自动裁剪空白区域，缩放到目标尺寸，主体居中 |
| R7 | Sprite Sheet 拼接 | 多帧素材拼成单张 Spritesheet + JSON 元数据（帧坐标/尺寸） |
| R8 | Tile 3×3 平铺预览 | 将单张 Tile 拼接为 3×3 网格，直观检查平铺效果 |
| R9 | Tile 边缘一致性评分 | 检测 Tile 四边 RGB 差异，输出 0-100 无缝评分 |
| R10 | ZIP 导出 | 打包素材 + Spritesheet + 元数据，支持 Unity / Godot 两种目录结构 |
| R11 | 前端交互界面 | 文本输入、参数选择、结果预览、导出下载的完整 Web UI |

### Should Have（P1 — 有加分）

| ID | 需求 | 说明 |
|----|------|------|
| R12 | 真实 AI API 对接 | 支持可配置的 AI Provider（OpenAI / Stability / 其他） |
| R13 | 素材历史记录 | 本地存储生成历史，可回溯 |
| R14 | 批量生成 | 一次生成多组素材 |

### Could Have（P2 — 锦上添花）

| ID | 需求 | 说明 |
|----|------|------|
| R15 | 调色板约束 | 限定生成素材的色板，确保风格统一 |
| R16 | 动画预览 | 在浏览器中播放 Sprite Sheet 帧动画 |

## 用户工作流

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  输入    │ -> │  AI 生成  │ -> │  后处理   │ -> │  预览    │ -> │  导出    │
│ prompt  │    │ (或DEMO) │    │ 透明/裁剪 │    │ 画廊/    │    │ ZIP 包   │
│ + 参数   │    │          │    │ 标准化/   │    │ Tile检查 │    │ 下载     │
│          │    │          │    │ 居中      │    │          │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

## 创新点

1. **资产化管线而非单纯生成**：从「生成的图像」到「引擎可用的资产」之间的所有处理环节全部自动化
2. **Tile 边缘一致性检测算法**：自行实现的图像边缘比对算法，填补 AI 生成 Tile 的质量评估空白
3. **DEMO 模式架构**：前后端均可无外部依赖运行完整流程，确保项目可复现、可评审
4. **引擎感知导出**：自动生成 Unity `.meta` 占位文件和 Godot `.import` 占位文件，资产导入即用
5. **渐进式 Provider 架构**：AI 生成层抽象为可替换 Provider，便于切换不同 AI 后端

## 非功能性需求

- **可复现**：评审者 clone 后无需任何 API key 即可跑通全流程
- **可评审**：每个 PR 小且聚焦，18 个 PR 渐进构建
- **主分支稳定**：main 分支任意时间点均可运行
- **代码规范**：前后端分别遵循 TypeScript strict mode 和 Python type hints
- **安全**：文件上传限制类型与大小，路径穿越防护

## 成功指标

- [ ] DEMO 模式下，从输入 prompt 到下载 ZIP 包全流程 < 30 秒
- [ ] 四类素材均可在 DEMO 模式下产出
- [ ] Tile 边缘评分对纯色 Tile 输出 ≥ 95 分
- [ ] 导出的 ZIP 包可被 Unity 和 Godot 直接识别
